# Context Match and Identity Match

TMP defines two operations. They are independent by design — processed in separate infrastructure, carrying different data, answering different questions. The publisher joins their results to make the final activation decision.

This page walks through both operations and the join using a concrete example.

## The Scenario

A retail publisher has three active AdCP media buys from a buyer agent. Each media buy contains packages:

- **Package A**: Sponsored products — coffee brands, carousel format, available on search and category pages
- **Package B**: Homepage takeover — premium display, available on the homepage only
- **Package C**: Seasonal promotion — iced beverages, native format, available site-wide during summer

A shopper searches for "best cold brew." The publisher's search results page loads.

## Context Match

The publisher sends a Context Match request to the TMP Router. The request contains the page context and the list of available packages. It contains no user identity.

### What the publisher sends

```json
{
  "request_id": "ctx-7f3a",
  "property_id": "retailer-web",
  "property_type": "website",
  "surface": "search_results",
  "content": {
    "domain": "shop.example.com",
    "content_hash": "a8f2e1",
    "topic_ids": ["retail.beverages.coffee", "retail.beverages.cold"],
    "language": "en",
    "content_rating": "safe"
  },
  "available_packages": [
    {
      "package_id": "pkg-A",
      "media_buy_id": "mb-coffee-q1",
      "format_ids": ["sponsored_carousel"],
      "catalog_ids": ["gtin-001", "gtin-002", "gtin-003", "gtin-004"]
    },
    {
      "package_id": "pkg-B",
      "media_buy_id": "mb-homepage",
      "format_ids": ["display_takeover"]
    },
    {
      "package_id": "pkg-C",
      "media_buy_id": "mb-summer-promo",
      "format_ids": ["native_text"],
      "catalog_ids": ["gtin-005", "gtin-006", "gtin-003"]
    }
  ]
}
```

Note what is absent: no user ID, no device information, no session token, no IP address. The buyer agent evaluates this request based purely on content context and package eligibility.

### What the buyer responds

The buyer agent evaluates each package against the context. Package B is homepage-only, so it does not match a search results page. Packages A and C match.

```json
{
  "request_id": "ctx-7f3a",
  "activate": [
    {
      "package_id": "pkg-A",
      "preferred_gtins": ["gtin-003", "gtin-001"],
      "preferred_tags": ["cold_brew", "iced"],
      "exclude_gtins": ["gtin-004"]
    },
    {
      "package_id": "pkg-C",
      "eligible_promotions": ["free-shipping-cold-brew"],
      "creative_variant": "summer-2026"
    }
  ],
  "signals": {
    "segments": ["coffee_enthusiast", "high_purchase_intent"],
    "targeting_kvs": {
      "category_affinity": "beverages",
      "seasonal_relevance": "high"
    }
  }
}
```

The buyer has done more than just say "on" or "off." For Package A, they specified which catalog items to prefer in the carousel (gtin-003 first, then gtin-001) and which to exclude (gtin-004 is out of stock). For Package C, they indicated an active promotion and selected a seasonal creative variant. They also returned enrichment signals — audience segments and targeting key-values — that the publisher can pass to their ad server.

### What Context Match never carries

- User IDs (hashed or otherwise)
- Device identifiers
- Session tokens
- IP addresses
- Any data that could identify a specific user

This is not a policy restriction. It is enforced by the Context Router TEE enclave, which processes Context Match requests in isolation and can be [attested](/docs/trusted-match/privacy-architecture) to confirm it has no access to identity data.

## Identity Match

Separately — and with a random time delay — the publisher sends an Identity Match request to the TMP Router. The request contains an opaque user token and the list of packages to evaluate. It contains no page context.

### What the publisher sends

```json
{
  "request_id": "id-9b2c",
  "user_token": "tok_hk82mfp1",
  "package_ids": ["pkg-A", "pkg-B", "pkg-C"]
}
```

The `user_token` is generated by the publisher. It is opaque to the buyer — the buyer can map it to their own identity graph (if they have a match), but cannot reverse it to PII or correlate it with any page context.

Note what is absent: no URL, no search query, no content signals, no topic IDs. The buyer agent evaluates this request based purely on user identity and package eligibility.

### What the buyer responds

```json
{
  "request_id": "id-9b2c",
  "eligibility": [
    {
      "package_id": "pkg-A",
      "eligible": true,
      "frequency_capped": false,
      "audience_match": true,
      "intent_score": 0.82
    },
    {
      "package_id": "pkg-B",
      "eligible": true,
      "frequency_capped": false,
      "audience_match": true,
      "intent_score": 0.65
    },
    {
      "package_id": "pkg-C",
      "eligible": false,
      "frequency_capped": true,
      "audience_match": true,
      "intent_score": 0.71,
      "catalog_eligibility": [
        { "gtin": "gtin-005", "eligible": false, "reason": "already_purchased" },
        { "gtin": "gtin-006", "eligible": true },
        { "gtin": "gtin-003", "eligible": true }
      ]
    }
  ]
}
```

The buyer reports that this user is eligible for Package A (not frequency-capped, matches the audience, high intent score). Package B is eligible too, but Context Match already excluded it for this page. Package C has hit its frequency cap for this user — and if the publisher chooses to override the cap, the buyer provides catalog-level eligibility: gtin-005 was already purchased, so it should be excluded from any carousel.

### What Identity Match never carries

- Page URLs
- Content hashes
- Search queries
- Topic IDs
- Content ratings
- Any data that could identify what the user is looking at

This is enforced by the Identity Router TEE enclave, which is a separate infrastructure component with no shared state with the Context Router.

## The Publisher-Side Join

The publisher now has two independent responses. They join them on their own infrastructure — neither the buyer agents nor the router see the joined result.

```
For each activated package from Context Match:

  pkg-A:
    Context says: activate, prefer gtin-003 and gtin-001, exclude gtin-004
    Identity says: eligible, not frequency-capped, audience match, intent 0.82
    → Activate pkg-A in sponsored carousel
    → Show gtin-003 first, then gtin-001
    → Apply "coffee_enthusiast" segment to ad server targeting

  pkg-C:
    Context says: activate, promotion "free-shipping-cold-brew", creative "summer-2026"
    Identity says: NOT eligible (frequency-capped)
    → Skip pkg-C for this user
    (If the publisher overrides the cap: filter gtin-005 from catalog since already purchased)

Result: Activate Package A only. Populate carousel with gtin-003 and gtin-001.
Set targeting KVs: category_affinity=beverages, seasonal_relevance=high.
```

The publisher already knows how to activate Package A — it maps to a sponsored carousel slot in their rendering system. TMP told them what to activate and how to configure it. The publisher handled the rest.

## Catalog Refinement

Catalog refinement is what separates TMP from a simple on/off switch. Buyers can express preferences within the bounds of each package:

**GTIN preferences**: An ordered list of catalog items the buyer wants featured. The publisher respects the ordering when populating carousels, search results, or recommendation slots.

**GTIN exclusions**: Items to suppress — out of stock, brand safety concerns, competitive conflicts.

**Creative variants**: Which version of the creative to serve — 15-second vs 30-second cutdown for CTV, dark-mode vs light-mode for web, language variants for international campaigns.

**Promotions**: Which promotions are currently active for this package — free shipping, percentage discounts, limited-time offers.

**Preferred tags**: Content tags that influence carousel ordering — "cold_brew" and "iced" for a summer campaign pushes cold beverages ahead of hot ones.

Refinement interacts with Identity Match. The buyer's GTIN preferences from Context Match are filtered through the catalog eligibility from Identity Match: items the user has already purchased, items in their cart, or items they've viewed repeatedly can be suppressed or deprioritized.

## Enrichment Signals

Context Match responses can include enrichment signals alongside package activation:

- **Segments**: Audience or contextual segments (e.g., "coffee_enthusiast", "high_purchase_intent") that flow into the publisher's ad server as targeting signals.
- **Targeting key-values**: Arbitrary key-value pairs (e.g., `category_affinity=beverages`) that the publisher can use for line item targeting, reporting breakdowns, or real-time decisioning.

Enrichment signals are additive — they are not tied to specific packages. A buyer agent might return enrichment signals even when it activates no packages, providing value as a data provider rather than a demand source.

This is how existing RTD (Real-Time Data) modules work today. TMP unifies enrichment and package activation into a single protocol, so a buyer agent can do both in one response.
