# Known spec ambiguities

This page enumerates **open spec ambiguities** that new AdCP implementers hit during conformance testing. Each item has a workaround so you can make progress while the underlying issue resolves.

Entries are removed as the upstream issues close. If you hit something not listed here that the [troubleshooting guide](/docs/building/implementation/storyboard-troubleshooting) also doesn't cover, file an issue on [adcontextprotocol/adcp](https://github.com/adcontextprotocol/adcp) and it will land here within a release.

## Classification

Each entry is tagged:

- **Tracked** â€” acknowledged gap, issue filed, workaround documented here.
- **Under review** â€” decision pending, interim behavior documented here.
- **Resolved** â€” decision landed but vectors/SDK may still lag; interim guidance applies until the lag clears, then the entry is removed.

## Response-envelope shape: `errors[]` vs `adcp_error`

**Status.** Under review â€” [#2587](https://github.com/adcontextprotocol/adcp/issues/2587).

**The ambiguity.** Some storyboards assert responses carry an `errors: [...]` array; older or transitional ones assert a single `adcp_error: {...}` object. Spec prose doesn't commit unambiguously to one shape.

**Interim guidance.** Return `errors: [...]` (array). It subsumes the single-error case and matches the newer storyboards. Track [#2587](https://github.com/adcontextprotocol/adcp/issues/2587) for the final normative commit.

**How it bites.** You implement one shape; half the vectors pass, the others fail with envelope-shape mismatch messages. Both implementations are defensible readings of the current spec; the fix is a spec decision, not an agent change.

## `past_start` handling: reject vs accept-and-adjust

**Status.** Resolved â€” [#2389](https://github.com/adcontextprotocol/adcp/pull/2389) (merged).

**Historical context.** The `schema_validation > past_start_date_reject` vector used to assert `error_code: INVALID_REQUEST`. Agents that accept-and-adjust past start times (also spec-conformant) silently failed the vector.

**Current state.** The storyboard was split into `past_start_reject_path` and `past_start_adjust_path`, both optional, both contributing to a `past_start_handled` flag. The final phase asserts at least one of the two contributed. Either behavior is now explicitly conformant.

**Why this still appears here.** Listed for the cross-reference. If you are on a repo state pre-[#2389](https://github.com/adcontextprotocol/adcp/pull/2389), upgrade; the old single-branch vector no longer exists on `main`.

## Re-cancel: `NOT_CANCELLABLE` vs `INVALID_STATE`

**Status.** Resolved at the vector level â€” [#2605](https://github.com/adcontextprotocol/adcp/issues/2605). A spec-side decision on whether to canonicalize one code remains open.

**The ambiguity.** A second cancel on a media buy that is already `canceled` satisfies two spec rules at once:
- [`media-buy/specification.mdx` Â§129, Â§254](https://github.com/adcontextprotocol/adcp/blob/main/docs/media-buy/specification.mdx): "Sales agents MUST reject updates to media buys in terminal states (`completed`, `rejected`, `canceled`) with error code `INVALID_STATE`"
- [`update_media_buy.mdx` Â§675](https://github.com/adcontextprotocol/adcp/blob/main/docs/media-buy/task-reference/update_media_buy.mdx): "`NOT_CANCELLABLE` â€” Media buy or package cannot be canceled"

Both codes are defensible for a re-cancel. Pick either.

**Current state.** The `invalid_transitions > second_cancel` vector now accepts either code via `allowed_values: ["NOT_CANCELLABLE", "INVALID_STATE"]`. Other error-code assertions across the storyboards were audited against the spec; no further single-branch narrowings of a documented `any_of` were found. Lists of rejected codes in other vectors (e.g. idempotency conflicts, validation failures, product lookup errors) already use `allowed_values`.

**Remaining ask (spec-side).** Pick one code as canonical for re-cancel â€” the ambiguity is a spec readability cost, not an interop problem. Tracked at [#2605](https://github.com/adcontextprotocol/adcp/issues/2605).

**Audit completeness.** The full audit covered every `check: error_code` in `static/compliance/source/`. Findings: one widening (above); the other single-branch assertions (`VERSION_UNSUPPORTED`, `GOVERNANCE_DENIED`, `INVALID_STATE` for pause/resume on canceled, `MEDIA_BUY_NOT_FOUND`, `PACKAGE_NOT_FOUND`, `TERMS_REJECTED`) are spec-locked to one code per canonical source and are not ambiguous. The `past_start` case remains resolved by [#2389](https://github.com/adcontextprotocol/adcp/pull/2389) (dual-path with `any_of` enforcement).

## Idempotency `missing_key` vector unreachable through SDK

**Status.** Tracked â€” [adcp#2607](https://github.com/adcontextprotocol/adcp/issues/2607) (spec), [adcp-client#678](https://github.com/adcontextprotocol/adcp-client/issues/678) (runner fix).

**The ambiguity.** The storyboard `idempotency > create_media_buy_missing_key` asserts that agents reject a mutating request missing `idempotency_key` with `INVALID_REQUEST` or `VALIDATION_ERROR`. But the reference `@adcp/client` SDK auto-injects `idempotency_key` client-side, so the vector body leaves the client with the key populated â€” the agent never sees the malformed request.

**Interim guidance.** Implement missing-key rejection in your agent anyway â€” the spec MUST still stands. Don't rely on the storyboard to verify it; unit-test directly against a raw-HTTP request. The planned fix (`probeRawRequest`, parallel to the existing `probeSignedRequest` for signing vectors) is tracked in [adcp-client#678](https://github.com/adcontextprotocol/adcp-client/issues/678) for 3.0 GA.

**How it bites.** The vector passes vacuously today â€” your agent appears conformant on this assertion regardless of whether it actually rejects missing-key inputs.

## Storyboards asserting fields not required by response schemas

**Status.** Tracked â€” [#2604](https://github.com/adcontextprotocol/adcp/issues/2604).

**The ambiguity.** Certain storyboards assert response fields (`list_id` echo, catalog counts, `pricing_options`, `vendor_cost`) that the corresponding response schemas mark as optional. An agent that omits the optional field is schema-conformant but vector-nonconformant.

**Interim guidance.** Include the optional fields the vectors assert even when your implementation would normally omit them. The assertion set is narrower than the full optional-field space â€” start by running the storyboards once, reading which optional fields the failing assertions reference, and adding those to your response builder.

**How it bites.** Schema validation passes, vector assertions fail, and the vector's error message points at a field you didn't realize was expected.

## `check_governance` conditional-approval response shape

**Status.** Tracked â€” [#2603](https://github.com/adcontextprotocol/adcp/issues/2603).

**The ambiguity.** The `check_governance` response can carry a `conditions` field when governance approves the buy conditionally, but the field's shape is not defined in the response schema. Implementers have variously returned `string[]`, `{id, description}[]`, or free-form strings.

**Interim guidance.** Return `conditions: [{id: string, description: string}]` â€” structured, explicit, and forward-compatible with a future `satisfied_at` or `satisfied_by` field. Track [#2603](https://github.com/adcontextprotocol/adcp/issues/2603).

**How it bites.** Vectors for conditional-approval scenarios may or may not assert on the `conditions` shape depending on when they were written. If your shape differs, vectors fail with opaque "field shape" messages.

## `$context.*` placeholder resolution rules

**Status.** Under review â€” [#2589](https://github.com/adcontextprotocol/adcp/issues/2589).

**The ambiguity.** Storyboards use `$context.correlation_id` placeholders that are resolved at vector-dispatch time from captured response fields. The resolution rules (what qualifies as a valid source, how to report an unresolvable placeholder, whether `$context.*` resolution failures are MUST-echo or MAY-echo in error responses) aren't fully normative.

**Interim guidance.** When your response participates in a context echo, echo every key from the request's `context` object back in the response. Don't filter context to a known-good subset. Error responses follow the same echo rule â€” include `context` even on error.

**How it bites.** Context echo failures cascade: a step that depends on a captured `$context.correlation_id` from an earlier step fails because the earlier step's response didn't carry `context` back. The error message surfaces at the dependent step, not at the true source.

## Error-code taxonomy drift

**Status.** Under review â€” [#2588](https://github.com/adcontextprotocol/adcp/issues/2588).

**The ambiguity.** Historical storyboards have used `INVALID_STATE`, `INVALID_TRANSITION`, and `INVALID_STATE_TRANSITION` for overlapping conditions. The lint at `scripts/lint-error-codes.cjs` tightens the canonical set, but some storyboards may still assert variants.

**Interim guidance.** Return `INVALID_STATE_TRANSITION` for invalid state transitions (pausing a completed buy, approving an archived creative, etc.). If a storyboard asserts a different variant, it's stale â€” `lint-error-codes.cjs` is the canonical source of which codes exist.

**How it bites.** You return the canonical code; a stale vector asserts the old variant; the vector fails with a misleading "wrong error code" message.

## Storyboard fixtures use hardcoded ids

**Status.** Tracked â€” [#2585](https://github.com/adcontextprotocol/adcp/issues/2585).

**The ambiguity.** Many storyboard `sample_request` blocks hardcode `product_id: "test-product"`, `pricing_option_id: "test-pricing"`, account domains, etc. If your sandbox doesn't happen to seed these exact ids, every dependent vector fails with `PRODUCT_NOT_FOUND` or similar.

**Interim guidance.** Seed the fixtures via `comply_test_controller.seed_product` (or your sandbox's equivalent) before the storyboards run. Two ids are load-bearing across many scenarios: `test-product`, `test-pricing`. Seed them first.

**How it bites.** Fails the very first vector. Easy to diagnose, boring to work around â€” but the first-time implementer doesn't know the fixture ids are shared across storyboards.

## Submitting new entries

When filing a new ambiguity, include:

- **What the spec says** â€” quote or link the ambiguous sentence.
- **What the vector asserts** â€” the specific `check:` and `value:`/`allowed_values:` from the YAML.
- **What you did** â€” the branch your implementation took.
- **Why both are defensible** â€” why your reading of the spec is not obviously wrong.

That triage is what separates "spec ambiguity" from "implementation bug." The first belongs here; the second belongs in [troubleshooting](/docs/building/implementation/storyboard-troubleshooting).
