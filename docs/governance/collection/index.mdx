Collection Governance enables program-level brand safety for advertising. While [property lists](/docs/governance/property/tasks/property_lists) control *where* ads run (apps, sites, CTV platforms), collection lists control *what content* ads run in or around — specific shows, series, podcasts, and other content programs.

Together, property lists and collection lists form the **inventory list** system: two parallel constructs that give buyers complete control over both the technical surfaces and the content programs that carry their advertising.

## Why collections need their own lists

A brand safety "do not air" list typically contains three kinds of exclusions:

| Exclusion type | Example | AdCP construct |
|---|---|---|
| Apps and platforms | "Not on this news app" | [Property list](/docs/governance/property/tasks/property_lists) |
| Specific programs | "Not in this crime drama" | **Collection list** |
| Contextual adjacency | "Not next to kids content, but G/PG animation OK" | [Content standards](/docs/governance/content-standards/index) |

Programs are independent of properties. A hit series airs on multiple streaming platforms, free ad-supported services, and cable syndication simultaneously. Excluding it means excluding a single logical entity distributed across many sellers and properties. Property lists can't express this — they operate on technical surfaces, not content.

## Three layers of brand safety

Collection lists sit between property lists and content standards in the brand safety stack:

| Layer | Construct | Abstraction | Evaluation time |
|---|---|---|---|
| Property | Property list | Where ads run (apps, sites) | Setup |
| Collection | Collection list | What content ads run in/around (shows, series) | Setup |
| Content | Content standards | What specific content is adjacent to the ad | Per-impression |

These three layers compose. A media buy can reference all three:
- **Property list**: approved CTV apps
- **Collection list**: excluded programs
- **Content standards**: contextual adjacency rules (nuanced policies like "exclude kids content except G/PG animation")

Most buyers will use one or two layers, not all three. A buyer who only needs to exclude specific programs can use a collection list alone. The three-layer model is a composition framework, not a requirement.

### When to use collection lists vs. content standards

Collection lists filter by declared metadata — genre tags, content ratings, distribution identifiers. They operate at setup time and apply uniformly.

Content standards evaluate individual artifacts (articles, episodes, pages) against a natural language policy with calibration exemplars. They handle nuance that metadata can't capture.

**Rule of thumb**: if it can be decided from the collection's declared metadata alone, it's a collection list filter. If it requires evaluating actual content, it's content standards.

## How collection lists work

### Distribution identifiers are the cross-publisher key

Collections in AdCP carry [distribution identifiers](/docs/media-buy/product-discovery/collections-and-installments) — platform-independent IDs like IMDb, Gracenote, and EIDR that uniquely identify a program regardless of where it airs. Collection lists use these identifiers to express "this program, everywhere."

**Gracenote ID guidance:** Use root-level IDs for collection lists — SH-prefixed for series, MV-prefixed for movies, SP-prefixed for sports programs. Do not use EP-prefixed episode IDs; episode-level exclusion is a content standards concern.

### Setup-time, not bid-time

Like property lists, collection lists are resolved once by the governance agent, cached by sellers, and used in delivery decisions without runtime calls. The governance agent notifies subscribers via webhook when the resolved list changes.

### Governance agents manage collection lists

The same governance agent that manages a brand's property list manages their collection list. Both are brand safety artifacts managed through the same CRUD pattern:

```json
{
  "tool": "create_collection_list",
  "arguments": {
    "name": "Nova Motors CTV Do Not Air — 2026",
    "base_collections": [
      {
        "selection_type": "distribution_ids",
        "identifiers": [
          { "type": "imdb_id", "value": "tt9999901" },
          { "type": "imdb_id", "value": "tt9999902" }
        ]
      }
    ],
    "filters": {
      "content_ratings_exclude": [
        { "system": "tv_parental", "rating": "TV-MA" }
      ],
      "genres_exclude": ["news"],
      "genre_taxonomy": "iab_content_3.0"
    },
    "brand": { "domain": "novamotors.com" }
  }
}
```

This list combines explicit program exclusions (by IMDb ID) with structural filters (no TV-MA, no news genre). The filters catch programs not individually listed — any new TV-MA series on any platform is automatically excluded.

## Targeting integration

Collection lists are referenced in targeting overlays alongside property lists. Two fields support both inclusion and exclusion:

```json
{
  "targeting": {
    "property_list": {
      "agent_url": "https://governance.pinnacleagency.com",
      "list_id": "pl_novamotors_approved_ctv"
    },
    "collection_list_exclude": {
      "agent_url": "https://governance.pinnacleagency.com",
      "list_id": "cl_novamotors_dna_2026"
    }
  }
}
```

| Field | Semantics | Use case |
|---|---|---|
| `collection_list` | Inclusion — only run in these collections | "Only buy pre-roll on these three shows" |
| `collection_list_exclude` | Exclusion — never run in these collections | Brand safety do-not-air lists |

A media buy can reference both simultaneously — "run in these approved shows, but never in these specific programs even if they appear on the approved list." The exclude list always wins on overlap.

For product-level collection targeting (seller declares which collections are available), use `collection_targeting_allowed: true` on products with collection selectors.

<Info>
**Why two fields for collections but one for properties?** Property lists predate the paired include/exclude pattern now used across other targeting dimensions (geo, audience, device). Collection lists follow the current pattern. A future evolution may add `property_list_exclude` for symmetry.
</Info>

## Seller resolution

When a seller receives a media buy with a collection list reference:

1. **Fetch and cache** the collection list from the governance agent
2. **Match entries** against their own collection inventory — by distribution identifiers (imdb_id, gracenote_id), collection_rid (if using the registry), or publisher_collections entries
3. **Apply filters** — content rating and genre filters from the list against their collection metadata
4. **Report coverage** — matched count, unmatched entries, and filtered count at buy creation time

## Relationship to property lists

Property lists and collection lists are sibling constructs — both are inventory lists managed by governance agents with the same lifecycle pattern (create, get, update, list, delete, webhook). They differ in what they address:

| Dimension | Property list | Collection list |
|---|---|---|
| What it identifies | Technical surfaces (domains, apps) | Content programs (shows, series) |
| Primary identifier | Property identifiers (domain, bundle ID) | Distribution identifiers (IMDb, Gracenote, EIDR) |
| Filters | Country, channel, property type, features | Content rating, genre, kind, production quality |
| Cache default | 24 hours | 168 hours (one week) |
| Cross-publisher | Via property registry (property_rid) | Via collection registry (collection_rid) |
