# check_governance

Seller-side governance webhook. The seller POSTs to the buyer's `governance_agent` URL when media buy events occur -- purchases, modifications, and delivery milestones. The governance agent validates against the campaign plan and returns approved/denied/conditions.

The governance agent maintains all state. The seller does not chain check IDs or track conversation history -- it posts what happened, and the governance agent correlates by `plan_id` + `media_buy_id`.

Governance checks cover the full media buy lifecycle through three phases:

| Phase | When | What's checked |
|-------|------|----------------|
| `purchase` | Before confirming `create_media_buy` | Budget, geo, channels, flight dates, policies |
| `modification` | Before confirming `update_media_buy` | Change magnitude, reallocation, new parameters |
| `delivery` | Periodically during delivery | Pacing, spend rate, geo drift, channel distribution |

## Implementation levels

Sellers can adopt governance checks incrementally:

- **Level 1: Purchase only** -- One outbound call per `create_media_buy`. The seller checks the initial purchase. This is the minimum viable integration.
- **Level 2: + Modification** -- One outbound call per `update_media_buy`. The seller checks changes before confirming them.
- **Level 3: + Delivery reporting** -- Periodic outbound calls during active delivery. The seller reports delivery metrics and the governance agent monitors for drift.

## When to call

The seller calls `check_governance` when processing a request on an account that has a `governance_agent` (set during [`sync_accounts`](/docs/accounts/tasks/sync_accounts)). The seller presents the authentication credentials from the `governance_agent` configuration -- the governance agent rejects calls without valid credentials.

### Purchase phase

The seller calls with `phase: "purchase"` (or omits `phase`, since it defaults to `purchase`) when processing a `create_media_buy` request:

1. Interprets the `create_media_buy` request and determines what it will actually deliver
2. Calls the `governance_agent` with the `plan_id` from the request and its `planned_delivery`
3. Proceeds, rejects, or adjusts based on the response

### Modification phase

The seller calls with `phase: "modification"` when processing an `update_media_buy` request:

1. Interprets the `update_media_buy` request and determines the new `planned_delivery`
2. POSTs to the `governance_agent` with `phase: "modification"`, the updated `planned_delivery`, and a `modification_summary`
3. Proceeds, rejects, or adjusts based on the response

### Delivery phase

The seller calls with `phase: "delivery"` periodically during active delivery:

1. Collects delivery metrics for the reporting period (spend, impressions, geo/channel distribution, pacing)
2. POSTs to the `governance_agent` with `phase: "delivery"`, `planned_delivery`, and `delivery_metrics`
3. If `approved`, continues delivery and schedules the next check at `next_check`
4. If `denied`, pauses delivery immediately and notifies the buyer
5. If `conditions`, adjusts delivery to meet conditions and re-checks immediately

The governance agent sets the cadence via `next_check` in the response. The seller SHOULD call no later than the `next_check` time. The governance agent MAY treat a missed deadline as a finding on the next delivery check.

The presence of `next_check` in a `purchase` phase response is the opt-in signal that the governance agent expects delivery reporting. If absent, no delivery reporting is expected.

## When the governance agent is unavailable

If the seller cannot reach the governance agent (timeout, network error), it SHOULD proceed with the media buy and log the failure. Governance checks should not block ad delivery when the governance infrastructure is unavailable. The seller SHOULD retry on the next scheduled check.

## Request

### Purchase

```json
{
  "tool": "check_governance",
  "arguments": {
    "plan_id": "plan_q1_2026_launch",
    "buyer_campaign_ref": "q1-2026-spring-launch",
    "seller": "https://seller.example.com",
    "media_buy_id": "mb_seller_456",
    "buyer_ref": "q1-launch-pinnacle-001",
    "phase": "purchase",
    "planned_delivery": {
      "geo": { "countries": ["US"] },
      "channels": ["olv"],
      "start_time": "2026-03-15T00:00:00Z",
      "end_time": "2026-06-15T00:00:00Z",
      "total_budget": 150000,
      "currency": "USD",
      "frequency_cap": { "max_impressions": 3, "per": "user", "window": { "interval": 1, "unit": "days" } },
      "audience_summary": "Adults 25-54, US, premium video inventory",
      "enforced_policies": ["us_coppa"]
    }
  }
}
```

## Response

### approved

```json
{
  "check_id": "chk_001",
  "status": "approved",
  "plan_id": "plan_q1_2026_launch",
  "buyer_campaign_ref": "q1-2026-spring-launch",
  "explanation": "Planned delivery is within plan parameters. Budget: $150,000 of $500,000 plan total. Geo: US (within plan). Channel: OLV (within 40-70% target range).",
  "expires_at": "2026-03-15T01:00:00Z"
}
```

The seller proceeds with the media buy.

### approved (purchase with delivery opt-in)

```json
{
  "check_id": "chk_001",
  "status": "approved",
  "plan_id": "plan_q1_2026_launch",
  "buyer_campaign_ref": "q1-2026-spring-launch",
  "explanation": "Planned delivery is within plan parameters. Budget: $150,000 of $500,000 plan total. Geo: US (within plan). Channel: OLV (within 40-70% target range).",
  "expires_at": "2026-03-15T01:00:00Z",
  "next_check": "2026-03-22T00:00:00Z"
}
```

The presence of `next_check` signals that the governance agent expects delivery reporting. The seller should begin sending `delivery` phase requests at the indicated time.

### approved (delivery phase)

```json
{
  "check_id": "chk_004",
  "status": "approved",
  "plan_id": "plan_q1_2026_launch",
  "buyer_campaign_ref": "q1-2026-spring-launch",
  "explanation": "Delivery on track. Week 1 spend: $12,500 of $150,000 (8.3%). Pacing is on target for 13-week flight. Geo and channel distribution match plan parameters.",
  "next_check": "2026-03-29T00:00:00Z"
}
```

The seller continues delivery and schedules the next governance check for the `next_check` time.

### denied

```json
{
  "check_id": "chk_002",
  "status": "denied",
  "plan_id": "plan_q1_2026_launch",
  "buyer_campaign_ref": "q1-2026-spring-launch",
  "explanation": "Planned delivery targets CA (Canada) which is not within the plan's geography.",
  "findings": [
    {
      "category_id": "strategic_alignment",
      "severity": "critical",
      "explanation": "Geo targeting includes CA but plan only covers US.",
      "details": {
        "plan_countries": ["US"],
        "planned_countries": ["US", "CA"]
      }
    }
  ]
}
```

The seller MUST NOT proceed with the media buy. It SHOULD return a `GOVERNANCE_DENIED` error to the buyer's orchestrator.

### denied (delivery phase -- geo drift)

```json
{
  "check_id": "chk_005",
  "status": "denied",
  "plan_id": "plan_q1_2026_launch",
  "buyer_campaign_ref": "q1-2026-spring-launch",
  "explanation": "Delivery has drifted outside plan parameters. 12% of impressions delivered in CA (Canada) which is not within the plan's geography.",
  "findings": [
    {
      "category_id": "strategic_alignment",
      "severity": "critical",
      "explanation": "Geo distribution shows 12% delivery in CA, but plan only covers US.",
      "details": {
        "plan_countries": ["US"],
        "actual_distribution": { "US": 88, "CA": 12 }
      }
    }
  ]
}
```

The seller MUST pause delivery immediately. It SHOULD notify the buyer's orchestrator and correct the geo targeting before resuming.

### conditions

```json
{
  "check_id": "chk_003",
  "status": "conditions",
  "plan_id": "plan_q1_2026_launch",
  "buyer_campaign_ref": "q1-2026-spring-launch",
  "explanation": "Budget approved but frequency cap must be applied per brand policy.",
  "conditions": [
    {
      "field": "frequency_cap",
      "required_value": { "max_impressions": 5, "per": "user", "window": { "interval": 1, "unit": "days" } },
      "reason": "Brand policy requires daily frequency cap of 5 or fewer impressions per user."
    }
  ],
  "expires_at": "2026-03-15T01:00:00Z"
}
```

The seller can either adjust its planned delivery to meet the conditions and re-check, or reject the media buy. The seller MUST NOT proceed without meeting the conditions.

Conditions with a `required_value` are machine-actionable -- the seller can programmatically apply the value. Conditions without a `required_value` are advisory -- the seller should interpret the `reason` and adjust accordingly.

### conditions (delivery phase -- overpacing)

```json
{
  "check_id": "chk_006",
  "status": "conditions",
  "plan_id": "plan_q1_2026_launch",
  "buyer_campaign_ref": "q1-2026-spring-launch",
  "explanation": "Delivery is pacing 40% ahead of schedule. Cumulative spend of $42,000 after 2 weeks exceeds expected $23,000 for this point in the flight.",
  "conditions": [
    {
      "field": "pacing",
      "reason": "Reduce daily spend rate to align with the planned flight duration. At current pace, budget will be exhausted by week 7 of 13."
    }
  ],
  "next_check": "2026-03-31T00:00:00Z"
}
```

The seller MUST adjust its pacing to meet the condition and re-check immediately. The `next_check` is set closer than normal so the governance agent can verify the correction.

## Fields

### Request

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `plan_id` | string | Yes | Campaign governance plan identifier from the `create_media_buy` request. |
| `buyer_campaign_ref` | string | Yes | Buyer's campaign reference from the `create_media_buy` request. |
| `seller` | string | Yes | URL of the seller agent requesting approval. |
| `media_buy_id` | string | Yes | The seller's identifier for the media buy being checked. |
| `buyer_ref` | string | Yes | The buyer's reference identifier from the `create_media_buy` request. |
| `phase` | enum | No | `purchase`, `modification`, or `delivery`. Defaults to `purchase` if omitted. |
| `planned_delivery` | object | Yes | What the seller will actually deliver. See [planned delivery](/docs/governance/campaign/specification#integration-with-create_media_buy). |
| `delivery_metrics` | object | Phase | MUST be present for `delivery` phase. Actual delivery performance data. |
| `modification_summary` | string | No | Human-readable summary of what changed. SHOULD be present for `modification` phase. |

### Delivery metrics

| Field | Type | Description |
|-------|------|-------------|
| `reporting_period` | object | Reporting window with `start` and `end` timestamps (ISO 8601). Required. |
| `spend` | number | Spend during the reporting period. |
| `cumulative_spend` | number | Total spend since the media buy started. |
| `impressions` | integer | Impressions during the reporting period. |
| `cumulative_impressions` | integer | Total impressions since the media buy started. |
| `geo_distribution` | object | Actual geographic distribution. Keys are ISO 3166-1 alpha-2 codes, values are percentages. |
| `channel_distribution` | object | Actual channel distribution. Keys are values from the channels enum, values are percentages. |
| `pacing` | enum | `ahead`, `on_track`, or `behind`. |

### Response

| Field | Type | Description |
|-------|------|-------------|
| `check_id` | string | Unique identifier for this governance check. For audit purposes only -- the seller does not need to store or reference this in subsequent calls. |
| `status` | enum | `approved`, `denied`, or `conditions`. |
| `plan_id` | string | Echoed from request. |
| `buyer_campaign_ref` | string | Echoed from request. |
| `explanation` | string | Human-readable explanation of the decision. |
| `findings` | array | Present when status is `denied` or `conditions`. Per-category issues found. |
| `conditions` | array | Present when status is `conditions`. Adjustments the seller must make. |
| `expires_at` | string | Present for `purchase` and `modification` phases when status is `approved` or `conditions`. The seller must act before this time. |
| `next_check` | string | When the seller should next report delivery metrics. Present in any phase when the governance agent expects ongoing delivery reporting. |

## Error codes

| Code | Recovery | Description |
|------|----------|-------------|
| `PLAN_NOT_FOUND` | correctable | No plan with this ID. The buyer may not have synced the plan yet. |
| `CAMPAIGN_SUSPENDED` | correctable | Campaign governance is suspended pending human review. |
| `SELLER_NOT_RECOGNIZED` | correctable | The seller URL is not in the plan's `approved_sellers` list. |
