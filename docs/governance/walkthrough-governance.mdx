An AI agent is about to spend $50,000 on advertising. No human reviewed the plan. No system checked the budget. No policy filtered the inventory. The agent has credentials, it has a brief, and it's ready to buy.

This is the "no eyes" problem. When agents operate autonomously, who makes sure they follow the rules?

This walkthrough shows how AdCP's governance protocol provides that oversight — not by slowing agents down, but by building a trust framework that lets them operate safely at speed.

## The three-party trust model

<img src="/images/concepts/three-party-governance.png" alt="Triangle diagram showing orchestrator proposes, governance validates, seller confirms" style={{ width: '100%', borderRadius: '12px', marginBottom: '1rem' }} />

AdCP governance separates duties across three independent parties:

| Party | Role | Cannot do |
|---|---|---|
| **Orchestrator** | Proposes campaign plans, executes buys | Cannot set its own spending limits or approve its own plans |
| **Governance agent** | Validates plans against policies, tracks budgets | Cannot execute buys or modify campaigns |
| **Seller** | Fulfills media buys, reports delivery | Cannot override governance decisions or modify budgets |

No single party controls the full workflow. This separation is what makes autonomous operation safe — the agent that spends money isn't the same agent that sets the rules.

## Step 1: Sync the plan

<img src="/images/walkthrough/governance-02-plan-synced.png" alt="A buying robot creates a campaign plan that floats across to a governance robot at a security desk" style={{ width: '100%', borderRadius: '12px', marginBottom: '1rem' }} />

Before executing any buy, the orchestrator registers its campaign plan with the governance agent:

```javascript
const plan = await governance.syncPlans({
  plans: [{
    plan_id: "acme-q2-trail-pro",
    brand: { name: "Acme Outdoor", domain: "acme-outdoor-example.com" },
    objectives: "Q2 Trail Pro 3000 launch across sports and outdoor lifestyle publishers",
    budget: { total: 50000, currency: "USD", authority_level: "agent_limited" },
    flight: { start: "2026-04-01T00:00:00Z", end: "2026-06-30T23:59:59Z" },
    countries: ["US", "CA"]
  }]
});
```

The governance agent now knows about this plan. It resolves applicable policies from the policy registry — brand safety rules, budget limits, regulatory requirements for US and Canada, and any advertiser-specific restrictions from Acme Outdoor's brand.json.

No money has moved. The plan is registered, not executed.

## Step 2: Check before spending

<img src="/images/walkthrough/governance-03-checks.png" alt="Governance robot reviewing three inspection panels — budget, brand safety, and compliance — with one showing an amber warning" style={{ width: '100%', borderRadius: '12px', marginBottom: '1rem' }} />

When the orchestrator is ready to buy, it calls `check_governance` before executing:

```javascript
const check = await governance.checkGovernance({
  plan_id: "acme-q2-trail-pro",
  buyer_campaign_ref: "acme-q2-ctv",
  binding: "proposed",
  caller: "https://orchestrator.pinnacle-agency.example",
  tool: "create_media_buy",
  phase: "purchase",
  payload: {
    seller: "https://streamhaus.example",
    amount: 25000,
    currency: "USD"
  }
});
```

The governance agent evaluates the proposed action against every applicable policy:

| Check | Status | Detail |
|---|---|---|
| Budget within plan limit | Passed | $25,000 of $50,000 available |
| Budget within agent authority | **Warning** | Agent authorized up to $20,000 per transaction |
| Brand safety property list | Passed | StreamHaus is on the approved list |
| Regulatory compliance | Passed | Targeting meets US/CA requirements |
| Creative provenance | Passed | All creatives carry required metadata |

The governance agent doesn't just return pass/fail. It returns structured **findings** with severity levels (`must`, `should`, `may`) and confidence scores. The orchestrator knows exactly what passed, what failed, and why.

## Step 3: Escalate when needed

<img src="/images/walkthrough/governance-04-escalation.png" alt="Governance robot raises an amber flag and routes the plan to a human reviewer at a glass desk" style={{ width: '100%', borderRadius: '12px', marginBottom: '1rem' }} />

The $25,000 transaction exceeds the agent's $20,000 authority limit. The governance agent flags this as a `must`-severity finding — the orchestrator cannot proceed without resolution.

The orchestrator has two options:
1. **Reduce the transaction** to $20,000 or less
2. **Escalate to a human** for approval

The orchestrator escalates. A campaign manager receives the flagged plan with full context — what the agent wants to buy, why it was flagged, and what policies apply.

## Step 4: Human approves with conditions

<img src="/images/walkthrough/governance-05-approved.png" alt="Human reviewer stamps the plan with approval and attaches a condition note, sending it back through the governance robot" style={{ width: '100%', borderRadius: '12px', marginBottom: '1rem' }} />

The campaign manager reviews the plan and approves it — but with a condition: the agent must report delivery weekly instead of at flight end.

This approval is recorded in the governance system. The governance agent updates the plan's delegation — the orchestrator now has temporary authority for this specific transaction, with the added reporting constraint.

The orchestrator proceeds with the buy. The governance agent records that a human approved it, who approved it, when, and under what conditions.

## Step 5: Campaign runs under watch

<img src="/images/walkthrough/governance-06-running.png" alt="Ads running across a cityscape while a governance robot monitors from a watchtower, scanning with a teal beam" style={{ width: '100%', borderRadius: '12px', marginBottom: '1rem' }} />

The campaign is live. Governance doesn't stop at the purchase — it monitors delivery against the approved plan:

- **Budget tracking**: As delivery reports come in via `report_plan_outcome`, the governance agent tracks actual spend against the committed budget
- **Drift detection**: If delivery diverges from the plan (wrong publisher, unexpected creative, budget overrun), the governance agent flags it
- **Policy updates**: If a new regulation takes effect mid-flight (e.g., a regional ad restriction), the governance agent applies it to active plans

```javascript
await governance.reportPlanOutcome({
  plan_id: "acme-q2-trail-pro",
  buyer_campaign_ref: "acme-q2-ctv",
  check_id: "chk-q2-ctv-001",
  outcome: "completed",
  seller_response: {
    media_buy_id: "mb-streamhaus-001",
    committed_budget: 24850
  },
  delivery: {
    media_buy_id: "mb-streamhaus-001",
    impressions: 887000,
    spend: 24850
  }
});
```

The governance agent records the actual outcome and updates budget availability. The remaining $25,150 is available for the next buy.

## Step 6: Full audit trail

<img src="/images/walkthrough/governance-07-audit-trail.png" alt="A timeline ribbon showing every decision point, with the governance robot presenting a detailed log to a team of humans" style={{ width: '100%', borderRadius: '12px', marginBottom: '1rem' }} />

At any point — during the campaign or years later — anyone can pull the complete decision history:

```javascript
const audit = await governance.getPlanAuditLogs({
  plan_id: "acme-q2-trail-pro"
});
```

The audit trail shows every event in sequence:

1. **Plan registered** — orchestrator synced plan with $50K budget
2. **Governance check** — $25K buy flagged for exceeding agent authority
3. **Human escalation** — campaign manager reviewed, approved with weekly reporting condition
4. **Buy executed** — StreamHaus media buy created
5. **Delivery reported** — $24,850 actual spend, 887K impressions
6. **Budget updated** — $25,150 remaining

Every decision, every approval, every outcome — structured, timestamped, and attributable.

## Crawl, walk, run

Not every team starts with full enforcement. AdCP governance supports three modes:

| Mode | Behavior | When to use |
|---|---|---|
| **Audit** | Log everything, block nothing | First deployment — see what governance would flag without disrupting workflows |
| **Advisory** | Warn on violations, don't block | Building confidence — review warnings and tune policies before enforcing |
| **Enforce** | Block violations, require resolution | Production — governance has teeth |

Start in audit mode. Review the logs. Tune your policies to reduce false positives. Move to advisory when you trust the signals. Move to enforce when you trust the system.

<Accordion title="Budget commitment phases">

<img src="/images/concepts/budget-commitment-phases.png" alt="Three-step diagram showing proposed (hypothetical), execute (reserved), and committed (final) budget states" style={{ width: '100%', borderRadius: '12px', marginBottom: '1rem' }} />

Budget tracking has three distinct phases:

1. **Proposed**: `check_governance` evaluates whether the amount fits within the plan. No money is reserved — this is a hypothetical check.
2. **Execute**: The seller runs the campaign. The governance agent tracks the authorized amount as reserved, but actual spend may differ.
3. **Committed**: `report_plan_outcome` records the actual amount. The governance agent updates the ledger with real numbers.

This matters because campaigns rarely spend exactly what was budgeted. A $25,000 buy might deliver $24,850. The governance agent tracks the difference and makes the remaining $150 available for future buys.

</Accordion>
