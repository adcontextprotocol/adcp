Enforce agent-hostname ownership at registration time. The agent URL's hostname must be exact-or-subdomain of an `organization_domains` row marked `verified = true` for the registering org — only verified rows count as a claim. `organizations.email_domain` is NOT consulted: it can be written from an unverified WorkOS domain via the brand-claim issue flow, so an attacker could otherwise claim `celtra.com` by issuing a DNS-pending challenge and registering a rogue agent under it (the exact escalation-#340 shape). Orgs without a verified-domain row hard-reject with `no_verified_domains` and must verify a domain via Linked Domains (Settings → Organization → Domains) first, or have an AAO admin register on their behalf via the cross-org admin endpoint from #4498. Catches the escalation-#340 failure mode at the registration boundary. Gates five write surfaces: `POST /api/me/agents`, `save_agent` MCP tool, `PUT /api/me/member-profile` (bulk profile update — entries already in the JSONB are grandfathered, new URLs go through the gate), `POST /api/me/member-profile` (create), and the visibility-flip routes (`POST/DELETE /agents/:index/publish`, `PATCH /agents/:index/visibility`) when target is `members_only` or `public` — a grandfathered row can't be promoted out of private without satisfying the gate. MVP of #4499 — adagents.json delegation and DNS challenge paths for legitimate cross-domain registration follow in later phases.
