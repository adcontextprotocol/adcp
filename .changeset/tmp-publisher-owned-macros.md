TMP: publisher-owned TMPX macro mapping and split provider→router / router→publisher response schemas.

Provider-to-router responses now carry `tmpx_values: [string]` — an ordered list of opaque URL-safe TMPX chunks. Publisher-local destination names (ad-server macros, GAM key-values, VAST URL macros, DOOH play-log fields) MUST NOT travel between provider and router. Router-to-publisher responses reshape `tmpx_providers[provider_id]` to `{ values: [string] }`, preserving the emitting provider's chunk order. Position within `values[]` is the chunk's identity; the publisher's deployment configuration (`tmpx_macro_mapping`, captured in the new `publisher-tmpx-config.json`) resolves each `(provider_id, index)` to the local destination on that surface. Providers no longer declare `tmpx_macros` on their registration entry — naming is a publisher-side deployment concern.

**Security-motivated reshape of an experimental surface.** Restores the direction described in #2203 by removing publisher-local names from the untrusted-provider boundary, and closes the cross-provider name-hijack surface tracked as #5945 by construction — the router never accepts destination names from providers.

**Why 3.1.x, not 3.2.** Trusted Match is `x-status: experimental`, which per `docs/reference/experimental-status.mdx` allows shape changes inside 3.x without a stable-surface deprecation cycle. Direct precedent: #5729 introduced the very `tmpx_macros`/`{name,value}` shape this changeset removes and landed as a patch on 3.1.x under the same rule (its changeset explicitly waived the 6-week notice window because the surface was hours old). #5729 shipped 26 days ago in 3.1.1–3.1.4; deferring this reshape to 3.2 while its opposite was patched into 3.1.x would leave the #5945 attack surface in every in-support release with no fix path. The change lives on the single-source schema tree so it rolls forward into 3.2 automatically — there is no 3.1.x fork to maintain.

Schemas: `trusted-match/identity-match-response.json` is now the router→publisher shape only (`tmpx_providers[provider_id].values`). New `trusted-match/provider-identity-match-response.json` captures the provider→router shape (`tmpx_values`). New `trusted-match/publisher-tmpx-config.json` captures the publisher-owned mapping shape. `trusted-match/provider-registration.json` drops the provider-declared `tmpx_macros` list. Spec narrative: IdentityMatchResponse field table splits per hop, a new `§Publisher TMPX macro mapping` section walks the configuration shape and adapter obligations (fail closed for that provider on that impression when the response carries more values than the mapping has slots), and the router-architecture and inventory-specific-behavior sections are rewritten to consume ordered values rather than provider-emitted names. Router-architecture also restates the outbound-envelope invariant (routers MUST drop any inbound `tmpx_values` at the outbound boundary rather than pass it through the extension surface).

**Migration.** Providers that emitted `tmpx_macros[{name,value}]` on their identity-match response emit `tmpx_values[value]` in the same order. Providers that registered a `tmpx_macros` list drop it from `provider-registration.json`. Routers that produced `tmpx_providers[provider_id].macros[{name,value}]` produce `tmpx_providers[provider_id].values[value]` in the same order. Publishers configure `tmpx_macro_mapping` in their deployment configuration to resolve each `(provider_id, index)` to the local destination. The legacy singular `tmpx` field remains supported through 3.x (removed in 4.0). Blast radius is bounded — the #5729 shape shipped only in 3.1.1–3.1.4 on an experimental field; adopters (if any) walk `values[]` and look up the destination in their publisher configuration instead of reading the wire-side `name`.
