Brand-claim `issueDomainChallenge` now surfaces `workos_misconfigured` (HTTP 503) when WorkOS returns a DNS-strategy domain without a `verification_prefix` — typically an env-wide config gap where the WorkOS DNS verification template isn't set up. Previously the service handed back a half-broken response and the route emitted `Publish the DNS TXT record at verification_prefix.{domain} ...` with the placeholder un-substituted, leaving users with no way to publish the record. The existing delete-and-recreate path is now scoped to genuinely-broken WorkOS state (missing `verificationToken`) — recreating won't fix a missing prefix, so the service stops churning tokens in a no-win loop and surfaces the env issue instead. The `/brand-claim/issue` route response also now includes a `dns_record_name` field and interpolates the verification prefix/token into the `instructions` string. Addie's `request_brand_domain_challenge` tool maps the new code to a stop-and-wait message rather than offering retries.
