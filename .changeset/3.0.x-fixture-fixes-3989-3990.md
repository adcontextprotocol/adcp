fix(compliance): storyboard fixture fixes — `inventory_list_targeting` sandbox routing + `sales_guaranteed/create_media_buy` task-completion path (mirror of #3989 / #3990)

Two storyboard-fixture-only fixes applied directly to the 3.0.x line, mirroring the same diffs filed against `main` in adcontextprotocol/adcp#3989 and adcontextprotocol/adcp#3990. Storyboard fixtures live in the compliance bundle that ships with each release; 3.0.x consumers running the worked-example seller (`hello_seller_adapter_guaranteed` from `@adcp/sdk` 6.7+) hit both bugs today.

**`inventory_list_targeting` — sandbox routing.** The 5 account blocks in this scenario use the brand+operator natural-key variant of `AccountReference` but omit the `sandbox` flag. Sellers whose `accounts.resolve` has separate code paths for sandbox vs production refs (the conventional shape) end up routing `create_media_buy` through the sandbox namespace and `get_media_buys` through the production namespace, because the SDK runner's `create_media_buy` enricher inherits `sandbox: true` from the test-kit while its `get_media_buys` enricher merges the storyboard's bare account block. Different `accountId`s break the framework's `mediaBuyStore` backfill, and `verify_create_persisted` / `verify_update_persisted` fail with `Expected ..., got undefined`. Setting `sandbox: true` on every account block keeps both create and get on the sandbox path. (Filed adcp-client#1487 to fix the underlying enricher asymmetry so future storyboard authors don't trip the same wire.)

**`sales_guaranteed/create_media_buy` — task-completion path.** The step exercises the spec-correct guaranteed-seller flow: returns an A2A task envelope (`status: 'submitted'`, no `media_buy_id` yet); the seller-assigned id only materializes on the eventual task-completion artifact after IO signing. The fixture's `context_outputs[0].path` was bare `"media_buy_id"`, which the runner resolves against the immediate submitted-arm response — where the field doesn't exist yet — producing `capture_path_not_resolvable` and breaking downstream phases that depend on `$context.media_buy_id`. The runner gained a `task_completion.` prefix in adcp-client#1426 (commit `3b21a15a`) for exactly this case: the prefix tells the runner to poll `tasks/get` for the terminal task artifact and resolve the rest of the path against the artifact data. Updating the path to `"task_completion.media_buy_id"` makes the fixture use the runner contract as designed.

Wire format unchanged on either fix. No protocol-version bump.
