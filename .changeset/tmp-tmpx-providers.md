TMP: provider-scoped TMPX macro trafficking. The router's identity-match fan-out reaches multiple identity providers, each of which may emit a TMPX exposure token (sometimes large enough to need more than one ad-server macro slot). The current single `tmpx` field collapses multi-provider responses to one string, loses attribution, and offers no shape for multi-chunk values.

Provider registrations now declare a stable, provider-namespaced list of ad-server macro names in `tmpx_macros` (e.g. `["S3_TMPX_1", "S3_TMPX_2"]`) — the names the publisher actually trafficks in GAM / VAST URLs / DOOH play logs. The identity-agent response emits `tmpx_macros[]` as ordered `{name, value}` pairs filling those slots. The router merges per-provider into `tmpx_providers: { provider_id: { macros: [{name, value}] } }` on the response so the publisher walks each provider's pairs and substitutes each `value` verbatim into the slot named by `name`. Multi-chunk values are capped at 2 per provider in v1; the cap MAY rise without a shape change. Macro names MUST NOT be derived from `provider_id` at runtime — trafficking is configured against the registered names ahead of time.

Schema updates: `tmp/provider-registration.json` adds `tmpx_macros`; `tmp/identity-match-response.json` adds `tmpx_macros` (provider-side) and `tmpx_providers` (router-merged), plus a shared `TmpxMacro` definition. Spec narrative: IdentityMatchResponse and ProviderEntry tables surface the new fields; `§Inventory-specific behavior` walks the per-macro substitution flow; `§Response Aggregation` in router-architecture.mdx adds a normative TMPX-collection paragraph that MUSTs the merge and explicitly forbids collapsing into a single string. The legacy single-token `tmpx` field is deprecated (still emitted by routers as a transitional convenience for consumers that haven't migrated; removed in 4.0).
