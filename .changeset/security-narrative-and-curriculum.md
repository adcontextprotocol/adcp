Weave security into the documentation narrative, retire the deprecated "principal" terminology, and tighten the idempotency scope to close a cross-account cache-collision vector.

- New **[Security Model](/docs/building/understanding/security-model)** narrative page for CISOs, security architects, and third-party risk reviewers — explains why agentic advertising raises the stakes (while preserving all traditional API security), walks through the 5-layer defense model (identity, isolation, idempotency, signed governance, auditability), and provides a pre-deployment checklist with data-handling and operational-readiness sections. Covers three agentic-specific threat classes (credential reuse across accounts, shared-governance-agent supply chain, prompt-injection credential exfiltration), describes AdCP's structural privacy separation (TMP Context/Identity split, signals license scoping, governance `policy_decision_hash`, client-side PII hashing), articulates where humans stay in the loop, and lists what AdCP explicitly does not do in 3.0.
- New section in **[Operating an Agent](/docs/building/operating-an-agent)** split into ad-ops health monitoring (task queues, lifecycle transitions, creative approvals, webhook delivery, status correctness) and security-as-operation (what compliance covers vs. what stays with you: credential storage, key ceremony, cross-instance and multi-region failover, security signal monitoring, vendor risk, incident response).
- **Idempotency scope tightened** in `docs/building/implementation/security.mdx` — from `(authenticated_agent, idempotency_key)` to `(authenticated_agent, account_id, idempotency_key)` everywhere, preventing cross-account cache collisions when one agent acts on multiple accounts. Added a normative entropy requirement for `account_id` values participating in cache scope (UUID/ULID, or hashed natural key) to close the existence-oracle inherited from the SI `session_id` pattern.
- **"Principal Isolation" section renamed to "Agent and Account Isolation"** with a two-step isolation pattern: auth precheck that request-supplied `account_id` is in the authenticated agent's authorized set, then a resource query scoped to that specific account. JWS buyer-identity-resolution wording clarified ("establish the buyer domain" rather than "establish the buyer agent"), and an explicit note added that the token's `iss` is untrusted until step 13 confirms it appears in the authenticated buyer's brand.json.
- **Curriculum sprinkles** in foundations (A1, A2, A3), specialist modules (S1 media buy, S4 governance), and Practitioner build projects (B4 publisher, C4 buyer), trimmed to stay within tier appropriateness: free-tier modules use plain-language framing; specialist modules walk the full resolution chain and the 15-step verification checklist.
- **Glossary additions**: Idempotency, Idempotency Key, SSRF. Principal entry kept with cross-links to Account and Agent as the migration signpost.
- **Terminology cleanup** across 22 files: normative signals/TMP/governance/architecture/accounts-overview/media-buy prose now uses agent + account per the long-standing glossary deprecation. Code samples and RLS policies rewritten to match the new isolation pattern.
- Filed **[issue #2369](https://github.com/adcontextprotocol/adcp/issues/2369)** proposing a dedicated S6: Security mastery specialist module with hands-on labs and learning objectives.
